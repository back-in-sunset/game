# User 服务增强任务清单（可执行版）

## 0. 说明

- 来源文档：
  - `code/common/service/user/docs/user-service-enhancement.md`
- 优先级定义：
  - `P0`：必须优先完成，直接影响安全与可用性
  - `P1`：重要能力，提升治理与运维效率
  - `P2`：平台化与长期优化
- 估时口径：
  - `S`：0.5-1 人日
  - `M`：1-3 人日
  - `L`：3-5 人日

---

## 1. P0 任务

### P0-1 忘记密码（发码+重置）
- 目标：
  - 支持未登录用户通过验证码重置密码。
- 改动点：
  - API：`/api/user/password/reset/send-code`、`/api/user/password/reset/confirm`
  - 逻辑：验证码校验 + 密码更新 + token 失效（若已实现 token version 则递增）
- 验收标准：
  - 验证码正确可重置，错误码语义清晰
  - 重置后旧密码不可用
- 估时：`M`

### P0-2 统一错误码体系
- 目标：
  - 替换散落字符串错误，统一为业务错误码。
- 改动点：
  - 新增 `errx` 包（API/RPC 通用错误码定义）
  - handler 返回统一结构
- 验收标准：
  - 登录、改密、改绑、绑定邮箱等关键接口都返回统一错误码
- 估时：`M`

### P0-3 邮箱/手机号校验与归一化
- 目标：
  - 防止脏数据进入主链路。
- 改动点：
  - 新增 `validator`（mobile/email）
  - 在发码、绑定、注册、修改接口统一应用
- 验收标准：
  - 非法格式均被拦截
  - 归一化后存储一致
- 估时：`S`

### P0-4 验证码审计表与落库
- 目标：
  - 可追踪验证码发送与校验行为。
- 改动点：
  - DDL：`user_verify_audit`
  - 在 `SendCode` 和 `VerifyCode` 路径写审计日志
- 验收标准：
  - 成功/失败/风控拒绝均可追踪
- 估时：`M`

### P0-5 事务一致性集成测试
- 目标：
  - 验证 `user`、`user_mobile_index`、`user_profile` 一致性。
- 改动点：
  - `integration` 测试：注册、改绑、删除、邮箱绑定并发冲突
- 验收标准：
  - 回滚路径无脏数据
- 估时：`M`

---

## 2. P1 任务

### P1-1 Token Refresh / Logout / Token Version
- 目标：
  - 完整会话生命周期管理。
- 改动点：
  - 新增 refresh token 接口
  - 用户表增加 `token_version`
  - 改密/重置时递增版本
- 验收标准：
  - 旧 token 能按预期失效
- 估时：`L`

### P1-2 用户状态管理（冻结/封禁）
- 目标：
  - 支持风控和运营处置。
- 改动点：
  - `user` 增加 `status/status_reason/status_expire_at`
  - 登录与敏感操作前置校验
- 验收标准：
  - 冻结账号无法登录和改绑
- 估时：`M`

### P1-3 管理端基础接口
- 目标：
  - 提供封禁、解禁、重置密码等运营能力。
- 改动点：
  - `/admin/user/freeze` `/admin/user/unfreeze` `/admin/user/reset-password`
- 验收标准：
  - 全部管理操作有审计日志
- 估时：`M`

### P1-4 邮箱唯一约束上线与数据清洗
- 目标：
  - 安全启用邮箱唯一索引。
- 改动点：
  - 编写清洗 SQL（重复非空邮箱）
  - migration + 回滚说明
- 验收标准：
  - 索引创建成功，无线上写入异常
- 估时：`S`

---

## 3. P2 任务

### P2-1 RiskEvaluator 平台化接入
- 目标：
  - 对接外部风控平台，不改业务逻辑。
- 改动点：
  - 定义 `RiskEvaluator` 接口
  - 实现 adapter（HTTP/gRPC）
  - 在登录、改绑、重置场景调用
- 验收标准：
  - 可按配置切换本地/平台实现
- 估时：`L`

### P2-2 验证码与风控监控大盘
- 目标：
  - 建立可观测性闭环。
- 改动点：
  - 指标：发送量、通过率、拒绝率、错误类型
  - 报警：短时失败激增、异常 IP/设备
- 验收标准：
  - 有监控图和告警策略
- 估时：`M`

### P2-3 RPC/API 协议收敛
- 目标：
  - 避免 API 独有逻辑难以复用。
- 改动点：
  - 扩展 `user.proto`：`UpdateProfile/ChangePassword/ChangeMobile/BindEmail`
  - API 层回归“薄层转发”
- 验收标准：
  - API 与 RPC 语义一致
- 估时：`M`

---

## 4. 依赖与顺序建议

1. 先做：`P0-2`（错误码）+ `P0-3`（校验）  
2. 再做：`P0-1`（重置密码）+ `P0-4`（验证码审计）  
3. 并行：`P0-5`（集成测试）  
4. 稳定后推进 `P1`，最后 `P2`
