# 平台模块架构说明

**产品名称**：Game Backend Cloud  
**文档版本**：v0.1  
**更新时间**：2026-04-27  
**文档目标**：说明平台模块分层、调用关系和责任边界，为后续微服务拆分与平台演进提供统一视图。  

---

## 1. 架构目标

平台架构设计要同时满足 4 个目标：

1. 支撑游戏实时服务的水平扩展
2. 支撑社区与 IM 的持续演进
3. 支撑多租户和多项目管理
4. 保证通用业务能力可复用

---

## 2. 总体分层

建议平台采用 6 层结构：

```text
客户端接入层
    ↓
接入网关层
    ↓
业务服务层
    ↓
实时服务层
    ↓
平台能力层
    ↓
基础设施层
```

---

## 3. 各层职责

## 3.1 客户端接入层

包括：

- 游戏客户端
- 社区客户端
- Web 后台
- 第三方接入 SDK

负责：

- 发起登录
- 访问社区接口
- 建立 IM 或实时连接
- 调用平台配置能力

这一层不直接感知底层服务实例，只通过统一入口访问。

## 3.2 接入网关层

包括：

- HTTP API 入口
- WebSocket/TCP 接入入口
- 认证与限流
- 租户/项目识别

负责：

- 身份认证
- 路由转发
- 灰度发布入口
- 审计与访问日志

长期来看，这一层可以承载统一网关能力，但当前阶段不要求全部落地。

## 3.3 业务服务层

包括：

- 用户服务
- 评论服务
- 动态服务
- 攻略服务
- 任务/道具/邮件等通用服务

负责：

- 提供标准业务 API
- 管理业务规则
- 提供稳定数据模型

这一层的核心原则是：通用业务尽量标准化，不夹带游戏实时逻辑。

## 3.4 实时服务层

包括：

- 匹配服务
- 房间服务
- 战斗服务
- IM 网关
- 状态同步服务

负责：

- 在线连接管理
- 实时消息交换
- 房间生命周期
- 弹性扩容

这一层是整个平台区别于普通社区 SaaS 的核心能力层。

## 3.5 平台能力层

包括：

- 租户管理
- 项目管理
- 环境配置
- 权限系统
- 配额系统
- 发布与运维入口

负责：

- 为上层业务提供 SaaS 化支撑
- 保证平台不是“单项目后端”，而是真正可复用平台

## 3.6 基础设施层

包括：

- MySQL
- Redis
- etcd
- DTM
- 对象存储
- 监控日志链路追踪组件

负责：

- 数据存储
- 配置发现
- 缓存与队列能力
- 可观测性

## 4. 模块调用关系

建议调用关系如下：

### 4.1 用户链路

```text
客户端
  → 网关
  → 用户 API
  → 用户 RPC
  → 用户模型 / MySQL / Redis
```

### 4.2 评论链路

```text
客户端
  → 网关
  → 评论 API
  → 评论 RPC
  → 评论模型 / MySQL / Redis
```

### 4.3 IM 链路

```text
客户端
  → IM 接入层
  → 连接管理
  → 消息路由
  → 存储 / 推送 / 离线处理
```

### 4.4 实时游戏链路

```text
客户端
  → 实时接入
  → 匹配
  → 房间服务
  → 战斗实例
  → 状态同步
```

## 5. 关键边界

## 5.1 业务服务与实时服务分离

不要把以下内容混在一个服务里：

- 用户/评论/动态等标准业务
- 匹配/房间/战斗等实时业务

原因：

- 扩容模式不同
- 状态模型不同
- 性能目标不同
- 生命周期不同

## 5.2 平台能力与业务能力分离

租户、项目、权限、配额这类能力不应散落在业务服务里，而应由平台层统一管理。

否则后面做 SaaS 化时会非常痛苦。

## 5.3 社区与 IM 解耦

社区内容和 IM 虽都属于社交层，但不建议耦成一个服务：

- 社区偏内容存储和检索
- IM 偏在线连接和消息分发

这两者的读写模型完全不同。

## 6. 当前仓库映射

当前仓库已经能够映射到以下层次：

- 业务服务层
  - `user`
  - `comment`
- 实时服务层
  - `im` 基础代码
- 基础设施层
  - `etcd/mysql/redis/dtm/prometheus/grafana/jaeger`

当前还明显缺失：

- 平台能力层
- 动态/攻略服务
- 匹配/房间/战斗服务
- 统一接入层

这说明平台还处于“单项目基础服务”向“SaaS 化平台”过渡的早期阶段。

---

## 7. 演进顺序建议

建议按以下顺序演进：

### Phase 1

- 稳定 `user/comment/im` 基础模块
- 补齐技术文档、任务清单、部署结构

### Phase 2

- 引入租户/项目/环境模型
- 引入平台管理后台
- 引入动态/攻略服务

### Phase 3

- 引入 IM 业务闭环
- 引入通知和系统消息

### Phase 4

- 引入匹配、房间、战斗层
- 引入水平扩容与调度体系
