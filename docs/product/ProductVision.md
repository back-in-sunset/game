# 产品总纲

**产品名称**：Game Backend Cloud  
**文档版本**：v0.1  
**更新时间**：2026-04-27  
**文档目标**：定义整个平台的产品形态、核心能力、租户模型和分阶段路线，作为后续各版本 PRD 的上位文档。  

---

## 1. 产品定义

Game Backend Cloud 是一个以游戏为核心的 SaaS 化后端平台，面向游戏工作室、独立开发团队和中小型发行团队，提供：

- 可水平扩展的游戏服务器能力
- 通用账号与业务系统
- 游戏社区能力
- IM 与通知能力
- 运营后台与部署运维能力

它的目标不是只服务某一款游戏，而是为多款游戏提供可复用、可配置、可托管的后端基础设施。

一句话定义：

`一个以游戏实时服务为核心，叠加社区、IM 与运营能力，并支持多租户和水平扩展的 SaaS 化游戏后端平台。`

---

## 2. 目标客户

### 2.1 独立开发者 / 小团队

诉求：

- 没有完整后端团队
- 希望快速上线 MVP
- 更关注接入速度和默认能力

需要的能力：

- 账号登录
- 评论和社区
- 基础 IM
- 简单运营后台

### 2.2 中小型游戏工作室

诉求：

- 已有游戏客户端
- 希望减少重复造轮子
- 希望支持弹性扩缩容

需要的能力：

- 房间服/战斗服扩容
- 多环境管理
- 日志监控
- 玩家社交和社区体系

### 2.3 发行 / 平台方

诉求：

- 同时管理多个游戏项目
- 统一运营、统一账号、统一社区

需要的能力：

- 多租户项目管理
- 多游戏接入
- 统一用户体系
- 统一消息与活动系统

## 3. 产品形态

平台整体拆成 4 层。

## 3.1 游戏运行层

负责游戏实例的实时运行和水平扩展：

- 网关服
- 大厅服
- 匹配服务
- 房间服
- 战斗服
- 状态同步 / 帧同步服务
- 服实例调度与弹性扩容

这是平台的核心差异化层。

## 3.2 通用业务层

负责游戏都需要的通用后端能力：

- 用户账号
- 鉴权
- 角色资料
- 背包 / 任务 / 邮件
- 排行榜
- 活动配置
- 支付与订单

这部分应尽量标准化、模板化。

## 3.3 社交内容层

负责社区与内容生态：

- 动态
- 攻略
- 评论
- 点赞 / 收藏
- 话题 / 标签
- 举报 / 审核
- 创作者激励

这部分决定产品不只是“游戏后端”，而是“游戏社区平台”。

## 3.4 实时通信层

负责玩家之间的实时互动：

- 单聊
- 群聊
- 频道
- 系统通知
- 在线状态
- 语音信令
- 后续实时语音能力

## 4. SaaS 化核心特征

如果要做成 SaaS，下面这些不是“可选项”，而是平台底座。

### 4.1 多租户

每个接入的游戏项目必须是独立租户。

租户维度需要隔离：

- 用户数据
- 游戏配置
- 社区内容
- IM 数据
- 存储资源
- 域名与接入配置
- 监控与日志视图

### 4.2 项目化管理

一个租户下可有多个项目，例如：

- 国服
- 国际服
- 测试服
- 活动服

项目级别应支持：

- 独立配置
- 独立环境
- 独立资源配额
- 独立发布策略

### 4.3 水平扩展

平台必须支持横向扩容，而不是仅靠单体扩机器：

- IM 网关按连接数扩容
- 匹配服务按请求量扩容
- 房间服/战斗服按房间数扩容
- 社区读服务按流量扩容

### 4.4 产品化接入

接入方式不能依赖“定制开发”，应提供：

- SDK
- OpenAPI
- Webhook
- 配置化后台

## 6. 平台能力边界

为了防止产品范围无限膨胀，需要明确边界。

平台负责：

- 后端基础设施
- 社交内容服务
- 通用业务能力
- 运维和运营能力

平台不直接负责：

- 游戏客户端开发
- 具体玩法逻辑设计
- 美术资源生产
- 复杂反外挂算法的完整落地

玩法相关逻辑应通过房间服/战斗服扩展能力承载，而不是全部内嵌到平台核心。

## 7. 产品套餐建议

## 7.1 基础版

面向独立开发者和小团队：

- 用户中心
- 鉴权
- 评论
- 动态
- 基础 IM
- 简单后台

## 7.2 实时版

在基础版上增加：

- 匹配
- 房间服
- 战斗服
- 扩容能力
- 在线状态与系统通知

## 7.3 社区版

在基础版上强化：

- 攻略
- 创作者体系
- 推荐
- 话题
- 内容审核

## 7.4 企业版

面向多项目和发行方：

- 多租户
- 多环境
- 配额管理
- 高级监控
- 私有化 / 专有化部署支持

## 8. MVP 路线建议

不建议一开始同时实现“全 SaaS 平台 + 全游戏实时体系 + 完整社区 + 完整 IM”。

更合理的路线是分三期。

## 8.1 Phase 1：SaaS 底座 + 基础业务

目标：

- 先形成平台雏形
- 让用户、评论、社区、基础部署真正可用

范围：

- 租户/项目基础模型
- 用户服务
- 评论服务
- 动态/攻略最小模型
- 基础后台
- 可组合中间件部署

## 8.2 Phase 2：IM 与社区强化

目标：

- 做出完整社交底层

范围：

- 单聊
- 群聊
- 系统通知
- 动态 feed
- 攻略与评论联动
- 内容审核

## 8.3 Phase 3：实时游戏运行层

目标：

- 拉开与普通社区产品的差异

范围：

- 匹配
- 房间服
- 战斗服
- 水平扩容
- 服调度

## 9. 当前仓库映射

基于当前仓库，已经有一定基础的模块：

- 用户服务：`code/common/service/user`
- 评论服务：`code/common/service/comment`
- IM 基础代码：`code/common/service/im`
- 部署拆分：`deploy/docker`

因此当前最现实的推进方式是：

1. 先把现有 `user/comment/im` 变成稳定基础模块
2. 再引入租户、项目和后台模型
3. 最后再建设实时游戏层
