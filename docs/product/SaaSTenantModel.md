# SaaS 租户与项目模型

**产品名称**：Game Backend Cloud  
**文档版本**：v0.1  
**更新时间**：2026-04-27  
**文档目标**：定义平台的多租户模型、项目层级、资源隔离与权限边界，为 SaaS 化落地提供统一约束。  

---

## 1. 设计目标

平台既然要服务多款游戏，就必须先把“谁在使用平台、谁拥有资源、谁管理环境、谁承担费用”定义清楚。

本模型解决 4 个问题：

1. 一个客户如何在平台上被识别
2. 一个客户如何管理多个游戏项目
3. 一个项目如何区分多个环境
4. 数据、配置、权限、资源如何隔离

---

## 2. 核心实体

建议平台采用 5 层核心实体：

- `Tenant`
- `Workspace`
- `Project`
- `Environment`
- `ServiceInstance`

## 2.1 Tenant

`Tenant` 是平台中的一级客户实体，通常代表：

- 一个游戏工作室
- 一个独立开发团队
- 一个发行公司

Tenant 拥有：

- 平台账号归属
- 计费主体
- 套餐与配额
- 数据归属
- 顶层权限域

一个 Tenant 下可以拥有多个 Workspace 和多个 Project。

## 2.2 Workspace

`Workspace` 是 Tenant 内部的协作空间，用来解决组织管理问题。

例如一个发行公司可以有：

- 研发工作区
- 运营工作区
- 海外工作区

Workspace 的作用：

- 划分成员范围
- 划分可见项目
- 划分后台视图

如果早期产品想简化，可以先不单独落库成复杂实体，但设计上应预留。

## 2.3 Project

`Project` 是平台中的核心业务单元，通常代表一款游戏或一个独立接入项目。

例如：

- `project_a`: 放置类手游
- `project_b`: 棋牌游戏
- `project_c`: 海外二次元社区游戏

Project 负责承载：

- 用户体系接入
- 游戏服务配置
- 社区内容配置
- IM 配置
- SDK / API 凭证

一个 Tenant 可有多个 Project。

## 2.4 Environment

`Environment` 是 Project 下的部署与运行环境。

常见环境包括：

- `dev`
- `test`
- `staging`
- `prod`
- `event`

Environment 负责隔离：

- 配置参数
- 中间件实例
- 数据库/缓存命名空间
- 域名或网关入口
- 发布版本

一个 Project 下至少应支持多个 Environment。

## 2.5 ServiceInstance

`ServiceInstance` 是具体运行中的服务实例或资源单元。

例如：

- 一个用户服务实例
- 一个 IM 网关实例
- 一个房间服实例
- 一个评论服务实例

它主要用于：

- 服务发现
- 水平扩展
- 弹性调度
- 监控与日志定位

---

## 3. 关系模型

建议关系如下：

```text
Tenant
  └── Workspace (optional)
        └── Project
              └── Environment
                    └── ServiceInstance
```

简化理解：

- Tenant 决定“是谁”
- Project 决定“做哪款游戏”
- Environment 决定“运行在哪个阶段”
- ServiceInstance 决定“当前有哪些服务实例在跑”

---

## 4. 隔离模型

SaaS 成败很大程度取决于隔离是否清晰。

## 4.1 数据隔离

至少应做到：

- 不同 Tenant 的数据逻辑隔离
- 不同 Project 的业务数据逻辑隔离
- 不同 Environment 的运行数据隔离

推荐分层：

- 租户级：逻辑隔离 + 访问控制
- 项目级：业务表命名空间或独立库
- 环境级：独立配置和缓存前缀

## 4.2 配置隔离

应按 `Tenant / Project / Environment` 三层区分：

- 平台级配置
- 租户级默认配置
- 项目级业务配置
- 环境级运行配置

## 4.3 资源隔离

需要控制以下资源：

- CPU / Memory
- 数据库存储
- Redis 容量
- 对象存储容量
- IM 连接数
- 在线用户数
- 房间服实例数

这部分是未来套餐和计费的基础。

## 4.4 权限隔离

需要至少支持：

- 平台管理员
- 租户管理员
- 项目管理员
- 开发成员
- 运营成员
- 只读成员

原则：

- 平台管理员可跨租户
- 租户管理员只能管理本租户
- 项目管理员只能管理被授权项目

---

## 5. 标识体系

建议全平台统一引入以下 ID：

- `tenant_id`
- `workspace_id`
- `project_id`
- `environment_id`
- `service_instance_id`

所有核心业务表、日志、指标、审计记录都应带上最少以下上下文：

- `tenant_id`
- `project_id`
- `environment_id`

否则后续无法真正做多租户追踪和计费。

---

## 6. 配额模型

建议配额至少分为 3 类：

## 6.1 平台资源配额

- 租户可创建项目数
- 项目可创建环境数
- 租户总成员数

## 6.2 运行资源配额

- 服务实例数上限
- 最大在线连接数
- 最大并发房间数
- 消息吞吐上限

## 6.3 存储配额

- 数据库存储量
- Redis 占用
- 媒体文件容量
- 日志保留时长

---

## 7. 计费模型建议

计费不一定一开始就实现，但租户模型必须为计费预留。

建议支持：

- 按套餐订阅
- 按资源使用量阶梯计费
- 按功能模块增购

可选计费维度：

- 项目数
- 在线用户数
- IM 长连接数
- 房间服运行时长
- 社区内容量
- 对象存储流量

---

## 8. 当前阶段最小落地版本

当前仓库还不需要一次完成完整 SaaS 平台，建议先落最小模型：

### Phase 1

- `Tenant`
- `Project`
- `Environment`

先不强制做：

- `Workspace`
- 完整计费
- 复杂配额引擎

Phase 1 的最小要求：

- 每个项目属于一个租户
- 每个项目至少支持 `dev/test/prod`
- 用户、评论、IM 配置都挂在项目和环境下

---

## 9. 对当前仓库的约束

接下来如果继续开发 `user/comment/im`，就需要开始考虑：

- 配置中预留 `tenant_id/project_id/environment`
- 后续表结构预留项目级隔离字段
- 后台和接口设计不要默认“全局只有一个项目”

这会影响后续很多设计决策，应该尽早统一。
