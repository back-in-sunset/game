# 配置中心与环境配置模型

**产品名称**：Game Backend Cloud  
**文档版本**：v0.1  
**更新时间**：2026-04-27  
**文档目标**：定义配置的层级、继承、覆盖、发布与回滚模型，为多项目、多环境和 SaaS 化运营提供统一规则。  

---

## 1. 设计目标

平台如果要支持多个租户、多个项目、多个环境，配置管理不能继续停留在“每个服务一个 yaml 文件”的阶段。

本模型要解决：

1. 配置属于谁
2. 配置在哪一层生效
3. 配置如何继承和覆盖
4. 配置如何发布、回滚和审计

---

## 2. 配置层级

建议采用 5 层配置体系：

- Platform Config
- Tenant Config
- Project Config
- Environment Config
- Runtime Override

## 2.1 Platform Config

平台全局默认配置。

示例：

- 默认登录策略
- 默认限流策略
- 默认日志保留时长
- 默认监控模板

适用范围：

- 全平台

## 2.2 Tenant Config

租户级配置，用来表达“这家公司/工作室的统一默认策略”。

示例：

- 租户默认语言
- 租户默认鉴权时长
- 租户默认社区审核策略

适用范围：

- 该租户下所有项目

## 2.3 Project Config

项目级配置，是业务最常用的一层。

示例：

- 项目名称、图标、域名
- 社区功能开关
- IM 功能开关
- 动态/评论/攻略模块开关
- 游戏服务接入参数

适用范围：

- 某个项目下的所有环境

## 2.4 Environment Config

环境级配置，用来表达同一项目在不同环境下的差异。

示例：

- 数据库连接
- Redis 连接
- 日志级别
- 测试开关
- 资源配额
- 发布镜像版本

适用范围：

- 指定环境

## 2.5 Runtime Override

运行时动态覆盖配置，用于短期变更。

示例：

- 临时降级开关
- 灰度比例
- 紧急限流值

特点：

- 生命周期短
- 必须可追踪
- 需要显式失效机制

---

## 3. 继承规则

建议配置优先级如下：

```text
Runtime Override
  > Environment Config
  > Project Config
  > Tenant Config
  > Platform Config
```

原则：

- 越具体的层级优先级越高
- 没设置时向上继承
- 禁止循环依赖和跨项目引用

---

## 4. 配置分类

建议把配置分成 4 类。

## 4.1 基础连接配置

例如：

- MySQL
- Redis
- etcd
- 对象存储

特点：

- 高敏感
- 生产风险高
- 需要权限保护

## 4.2 业务功能配置

例如：

- 评论开关
- 动态开关
- 攻略开关
- IM 开关
- 审核模式

特点：

- 运营、产品、项目管理员可能都会使用

## 4.3 运行参数配置

例如：

- 并发阈值
- 限流值
- 缓存 TTL
- 房间服最大数量

特点：

- 影响性能和容量

## 4.4 展示与接入配置

例如：

- 域名
- SDK Key
- 接口签名配置
- 品牌文案

---

## 5. 发布模型

配置不能只允许“直接改”，需要有发布概念。

建议采用：

- 草稿
- 待发布
- 已发布
- 已回滚

### 5.1 草稿

用户编辑中的配置，不立即生效。

### 5.2 待发布

已经完成编辑，等待审批或确认发布。

### 5.3 已发布

当前线上生效版本。

### 5.4 已回滚

历史版本恢复后形成的新生效版本。

---

## 6. 版本模型

每份配置都建议具备：

- `config_id`
- `scope_type`
- `scope_id`
- `version`
- `status`
- `created_by`
- `published_by`
- `change_log`

目标：

- 看得见谁改了什么
- 能对比版本
- 能回滚

---

## 7. 配置发布策略

建议支持：

- 立即发布
- 定时发布
- 按环境发布
- 灰度发布

最小阶段先实现：

- 按环境发布
- 版本回滚

---

## 8. 与权限模型的关系

配置系统必须和权限模型联动。

建议：

- Project Admin：可改项目级业务配置
- Project Developer：可改开发/测试环境配置
- Project Operator：可改部分功能开关，但不可改基础连接配置
- Environment Maintainer：可发布或回滚指定环境配置

特别是以下配置必须严格控制：

- 生产数据库连接
- Redis 连接
- SDK 密钥
- 签名私钥

---

## 9. 当前阶段最小落地版本

当前仓库实际仍以本地配置文件为主，因此建议分阶段推进。

### Phase 1

先统一配置模型，不急着做完整配置中心：

- 项目级配置
- 环境级配置
- 配置版本号
- 配置变更记录

### Phase 2

再逐步平台化：

- 租户级默认配置
- 可视化配置后台
- 发布与回滚

### Phase 3

最后完善高级能力：

- 灰度配置
- 动态覆盖
- 审批流

---

## 10. 对当前仓库的影响

后续继续开发时，建议遵守以下约束：

- 配置结构尽量区分项目和环境
- 不要把所有配置都写死在单份 yaml
- 新增服务时就考虑环境差异

否则等到真正做 SaaS 配置中心时，迁移成本会很高。
