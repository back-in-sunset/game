# 权限与角色模型

**产品名称**：Game Backend Cloud  
**文档版本**：v0.1  
**更新时间**：2026-04-27  
**文档目标**：定义平台的角色体系、权限边界和授权范围，为多租户后台、项目协作和运维操作提供统一模型。  

---

## 1. 设计目标

权限模型需要解决 3 个问题：

1. 谁能看到哪些资源
2. 谁能执行哪些操作
3. 谁能在什么范围内授权别人

平台是 SaaS 化产品，权限不能只做“管理员/普通用户”两档，至少要覆盖：

- 平台级
- 租户级
- 项目级
- 环境级

---

## 2. 设计原则

### 2.1 范围优先

同一个角色，在不同范围内权限不同。

例如：

- 项目管理员只能管理被授权项目
- 租户管理员不能操作其他租户

### 2.2 最小权限

默认只授予完成工作所需最小权限：

- 研发不默认拥有生产删改权限
- 运营不默认拥有租户级配置权限

### 2.3 显式授权

所有跨项目、跨环境、跨租户的高风险操作都必须是显式授权，而不是隐式继承。

### 2.4 操作可审计

以下操作必须可审计：

- 角色变更
- 权限授予/撤销
- 生产环境配置修改
- 封禁、删帖、内容审核
- 发布与回滚

---

## 3. 角色层级

建议采用 4 层角色体系：

- 平台角色
- 租户角色
- 项目角色
- 环境角色

## 3.1 平台角色

### Platform Owner

权限：

- 管理所有租户
- 管理平台级配置
- 管理套餐与配额规则
- 管理全局审计与监控

适用人群：

- 平台核心运维
- 平台超级管理员

### Platform Admin

权限：

- 查看和维护所有租户
- 支持租户协助排障
- 查看平台级日志和指标

限制：

- 不一定拥有计费和套餐设计权限

### Platform Support

权限：

- 只读查看租户和项目信息
- 查看日志、告警、状态
- 协助工单排查

限制：

- 默认不能修改租户生产配置

## 3.2 租户角色

### Tenant Owner

权限：

- 管理本租户所有项目
- 管理本租户成员
- 管理本租户套餐和资源分配
- 配置租户级默认策略

### Tenant Admin

权限：

- 管理本租户项目
- 管理成员和角色
- 查看账单、用量、告警

限制：

- 不一定拥有租户所有权转移能力

### Tenant Finance

权限：

- 查看账单
- 查看套餐
- 管理支付相关信息

限制：

- 不应拥有项目配置和生产操作权限

### Tenant Auditor

权限：

- 只读查看本租户审计日志、成员、项目、资源用量

## 3.3 项目角色

### Project Owner

权限：

- 管理项目配置
- 管理项目成员
- 管理项目环境访问
- 查看项目所有业务数据面板

### Project Admin

权限：

- 管理项目业务模块
- 管理项目级 API 密钥
- 管理项目社区与 IM 开关

限制：

- 不一定能管理租户层设置

### Project Developer

权限：

- 查看开发/测试环境配置
- 发布测试环境服务
- 查看日志与监控

限制：

- 默认不能修改生产环境高风险配置

### Project Operator

权限：

- 管理动态、攻略、评论审核
- 发布公告、活动
- 管理用户封禁、举报、推荐

限制：

- 不应拥有底层服务发布权限

### Project Analyst

权限：

- 查看项目报表、运营数据、留存数据、内容数据

限制：

- 默认只读

## 3.4 环境角色

环境角色是对高风险环境的补充授权。

### Environment Maintainer

权限：

- 修改指定环境配置
- 触发发布
- 回滚指定环境

### Environment Observer

权限：

- 查看该环境日志、监控、追踪、资源状态

环境角色通常只对：

- `staging`
- `prod`

这种高风险环境有意义。

---

## 4. 授权范围模型

每条授权建议至少包含：

- `subject_id`
- `subject_type`
- `role`
- `scope_type`
- `scope_id`

其中：

- `subject_type`：用户 / 组 / 机器人账号
- `scope_type`：platform / tenant / project / environment

示例：

```text
user_1001 -> ProjectDeveloper -> project_2001
user_1002 -> ProjectOperator -> project_2001
user_1003 -> EnvironmentMaintainer -> project_2001:prod
```

---

## 5. 权限域拆分

建议把权限拆成如下域：

- 成员管理
- 配置管理
- 发布管理
- 运行观测
- 内容审核
- 用户运营
- 账单与套餐
- API 密钥管理

这样角色可以由多个权限域组合，而不是完全写死。

---

## 6. 高风险操作清单

以下操作必须额外保护：

- 删除项目
- 删除生产环境
- 修改生产环境连接配置
- 回滚线上服务
- 批量封禁用户
- 批量删除社区内容
- 导出全量用户数据

建议增加：

- 二次确认
- 审批流
- 审计日志
- 操作理由

---

## 7. 当前阶段最小落地版本

当前阶段不需要一次完成完整 RBAC 平台，建议先落：

- Platform Admin
- Tenant Owner
- Project Admin
- Project Developer
- Project Operator
- ReadOnly

最小要求：

- 项目维度可授权
- 运营和研发角色分离
- 生产环境修改权限单独控制

---

## 8. 对当前仓库的影响

后续如果开始做后台或平台层，需要避免以下错误：

- 默认所有人都能看所有项目
- 默认研发和运营共用一套管理员权限
- 不区分测试环境和生产环境的操作权限

这些问题如果前期不控制，后面补权限会代价很高。
